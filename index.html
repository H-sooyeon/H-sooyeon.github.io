Next-auth 통합 인증 설계 → Sliding Session 도입

환경 간 데이터 단절 해결 · 동시성 제어 · 세션 만료 UX 개선

Part 1 · Next-auth 통합 인증 설계
문제
  • 클라이언트 메모리 기반 Access Token 관리 → 서버 컴포넌트에서 토큰 접근 불가
  • SSR 시 인증 데이터 패칭이 불가능한 결함 발견
해결
  • Next-auth v5 도입 — 세션 정보를 암호화된 쿠키로 관리해 서버·클라이언트 어디서든 최신 세션 접근 가능한 통합 인증 인터페이스 구축 PR #138
  • 서버 컴포넌트 병렬 요청 시 발생하는 RTR(Refresh Token Rotation) reuse detected 에러 → JWT 콜백 기반 갱신 로직 일원화 및 백엔드 Grace Period 도입 제안·적용 PR #145
  • 런타임 환경에 무관하게 최신 인증 헤더를 자동 주입하는 단일 fetchApi 유틸 구축 → 코드 응집도 향상 및 보안 강화
결과
  • 서버·클라이언트 컴포넌트 간 인증 상태 동기화 — 데이터 단절 문제 해결
  • 병렬 요청 환경에서의 동시성 문제 제어
Part 2 · Sliding Session 도입
문제
  • Refresh Token 만료로 인해 사용 중 갑작스러운 로그아웃 발생
  • 기록 작성 중 세션 만료 → 작성 데이터 유실
  • 사용자 피드백: "로그인이 너무 자주 풀린다"
비교 검토
기존 방식 유지
구현 단순
UX 저하, 재로그인 빈번
Access Token만 재발급
서버 부담 적음
Refresh Token 만료 문제 근본 해결 불가
선택
Sliding Session
활동 기반 세션 연장 → UX 개선
토큰 관리 복잡도 증가
선택 이유
서비스 특성상 "기록 작성 중 끊김"은 치명적인 UX 문제이므로, 보안 복잡도 증가보다 UX 저하 리스크가 더 크다고 판단했습니다. Discussions #272
해결
  • Refresh Token 만료 7일 이전 자동 재발급 로직 구현
  • Access Token 재발급 시 Refresh Token도 함께 갱신
결과
  • 로그인 유지 경험 개선
  • 작업 중 세션 만료로 인한 데이터 유실 방지
  • 사용자 피드백 기반 UX 개선 성공
화면 / 시연
🖼 인증 플로우 다이어그램
▶ 슬라이딩 세션 시연 영상
배운 점
인증은 단순 보안 로직이 아니라 UX 설계 요소입니다. 사용자의 행동 패턴을 고려한 설계가 얼마나 중요한지를 직접 경험했습니다.

이미지 로딩 구조 개선과 보안 강화

Proxy API + 캐싱 전략 도입 및 코드 리뷰를 통한 보안 개선

📝 블로그 PR #264
문제
  • Presigned URL 방식 → 캐싱 불가
  • 이미지마다 개별 요청 발생 → N+1 문제
  • 메인 스레드 블로킹 발생
비교 검토
클라이언트 fetch 유지
구조 단순
성능 저하, 캐싱 불가
Batch API 도입
요청 수 감소
Presigned URL 특성상 캐시 불가
선택
Proxy API + 캐싱
캐싱 가능, 성능 개선
서버 부하 증가
선택 이유
요청 수 감소보다 캐시 히트율이 더 중요하다고 판단했습니다. 재방문 사용자 경험 개선을 우선시했습니다.
해결 — 1차
  • /api/media-image/[id] 기반 Proxy API 설계
  • Sharp를 활용한 WebP 자동 변환
  • Cache-Control: immutable 적용
추가 발견 (코드 리뷰)
  • 캐시된 이미지가 인증 없이 외부에 노출될 가능성 발견
최종 개선 — 보안 강화
  • 이미지 데이터를 민감 정보로 재정의
  • ETag + no-cache 전략 적용
  • 매 요청마다 인증 검증 + 변경 없으면 304 응답으로 트래픽 최소화
결과
  • 이미지 렌더링 속도 개선
  • 캐시 효율 증가
  • 인증 우회 보안 리스크 제거
화면 / 시연
🖼 Proxy API 구조 다이어그램
📊 로딩 속도 개선 전/후 비교
배운 점
성능 최적화는 항상 보안과 트레이드오프 관계입니다. 코드 리뷰는 단순한 수정 과정이 아니라 설계를 개선하는 기회임을 배웠습니다.

PWA 한계 개선 및 Capacitor 기반 모바일 앱 확장

브라우저 제약 극복 → 하이브리드 앱 전환

Part 1 · PWA 구축 및 한계 개선
문제
  • 빌드 타임에 생성되는 manifest.json만으로는 다크/라이트 테마 전환 시 시스템 UI(주소창) 색상 실시간 동기화 불가
  • 브라우저별로 상이한 PWA 설치 프로세스로 인한 사용자 혼란 예측 ADR
해결
  • 테마 깜빡임 방지 — JS 로드 전 HTML 헤더 단계에서 localStorage를 확인해 메타 태그를 즉시 수정하는 인라인 스크립트 삽입 → 앱 실행 시 테마 색상이 뒤늦게 적용되는 시각적 노이즈 제거
  • 사용자 주도 설치 유도 — beforeinstallprompt 이벤트를 캡처해 서비스 내 설치 배너 구현, 해당 이벤트를 미지원하는 브라우저(Safari, Vivaldi 등)를 위한 커스텀 안내 모달 별도 제작
  • display-mode: standalone 감지로 중복 배너 노출 제어 → 네이티브 앱에 가까운 사용 경험 제공
Part 2 · Capacitor 기반 앱 확장
문제
  • 웹 서비스만으로는 앱 스토어 배포 불가
  • 모바일 환경에서 UX 품질 부족
비교 검토
PWA 유지

브라우저별 제약, UX 한계
네이티브 앱 개발

비용과 시간 부담 큼
선택
Capacitor 하이브리드

제한된 리소스 내에서 빠른 확장이 필요했으며, 기존 웹 자산을 최대한 활용하는 것이 가장 효율적인 전략이었습니다.
해결

  • SNS 로그인 후 앱 복귀 처리 구현

  • 상태바 / 인디케이터 영역 처리 및 웹·네이티브 UI 일관성 유지
결과

동일한 코드베이스라도 웹과 앱은 전혀 다른 UX를 요구합니다. 플랫폼 특성을 깊이 이해하지 않으면 기능은 동작하지만 사용감이 어색한 결과가 나온다는 것을 직접 체험했습니다.

📝 ADR
문제
  • Zustand 사용 시 페이지 이탈 후에도 데이터가 잔존
  • 전역 상태 오염으로 인한 예기치 않은 동작 가능성
개선
  • Context API 기반 페이지 단위 상태 관리 구조로 전환
  • 컴포넌트 마운트/언마운트 사이클에 맞춰 상태 생명주기 자동 관리

  • 페이지 이탈 시 상태 자동 초기화 — 데이터 잔존 문제 해소

🗂 상태 구조 변경 전/후 다이어그램
